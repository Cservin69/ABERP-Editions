### F1 🔴 Storefront accepts 11 CAD extensions; extractor handles 3; classifier doesn't catch the mismatch

- **Target session:** **S297 sweep**. Fix (1) is one line in `ALLOWED_EXT`; fix (2) is two lines in `classify_failure` plus a pin test mirroring `pr273_classify_step_assembly_rejection_is_permanent`.

### F2 🔴 STEP extractor does not validate or normalise length units

- **Evidence:** `python/aberp-cad-extract/aberp_cad_extract/extractors/step.py:139-180` reads the shape via `STEPControl_Reader.ReadFile` → `TransferRoots` → `OneShape`, then runs `BRepBndLib.AddOptimal_s` + `BRepGProp.VolumeProperties_s` on the raw shape. There is no `Interface_Static_SetCVal("xstep.cascade.unit", "MM")` call, no `UnitsAPI` invocation, no post-transfer unit inspection. The pinned fixtures (`tests/fixtures/unit_cube.step`, `thin_plate.step`, `assembly_two_solids.step`) were generated via OCCT (per the S292 memo's "regeneration helper") and inherit whatever unit OCCT defaulted to — there is no test of a METRE-declared or INCH-declared file. The wrapper crate test `crates/aberp-cad-extract-wrapper/tests/step_extract_smoke.rs` asserts `bbox == 20 mm`, but only against the OCCT-generated fixture; it does not exercise the unit path.
- **Why it matters:** real customer STEP files routinely declare `LENGTH_UNIT('METRE')` — SolidWorks defaults to mm but Fusion's "Save as STEP" can emit metres if the document units are set that way, and downstream CAD-CAM exports vary. A METRE-declared 20 mm cube reads back as a 0.020 mm cube. The engine's `clamp non-positive` defence (per S268 memo) passes a tiny-positive volume; the quote-engine produces a near-zero machining cost; `material_cost = volume_mm3 / 1000 * density * grade_price_per_kg`. The daemon emits a `Posted` row, the priced PDF lists "0,02 € total", the customer types ACCEPT, ABERP runs DEAL, books a Sales Order at a few cents — silent value loss is the most expensive class of bug (CLAUDE.md rule 12, fail loud).
- **Recommended fix:** at the top of `extract_step`, before `_load_step_shape`, call:
  ```python
  from OCP.Interface import Interface_Static
  Interface_Static.SetCVal_s("xstep.cascade.unit", "MM")
  ```
  This makes OCCT normalise to mm during `TransferRoots`. Pair with a pin test that loads a fixture declaring `LENGTH_UNIT('CENTI METRE')` or `'INCH'` and asserts bbox is in mm. If a fixture cannot be authored, an integration test that hand-edits the LENGTH_UNIT header line in a copy of `unit_cube.step` and asserts the post-transfer bbox is still 20.0 mm is sufficient.
- **Target session:** **S297 sweep — load-bearing for first real customer run.** If S297 cannot land this week, the storefront walkthrough Step 1 needs an explicit warning that "STEP file units MUST be mm" with no protection in code (operator-discipline trap, but better than silent wrong quotes).

### F3 🔴 Status state-machine forbidden-transition guard pre-empts idempotent same-state

- **Evidence:** `ABERP-site/src/routes/api/quotes/[id]/status/+server.ts:41-103`. Execution order:
  1. line 44: `if (to === 'quoted') return forbidden`
  2. line 51: `if (to === 'approved') return forbidden`
  3. line 67: `if (from === to) return ok(noop)`

  Consequently `POST /api/quotes/{id}/status` with `{status: 'approved'}` against a row already in `approved` returns 403, not a noop 200. Same for `quoted`. By contrast `quoting → quoting` and `invoiced → invoiced` correctly noop because they don't hit a forbidden-`to` gate first.
- **Why it matters:** the ABERP-side intake daemon (per the walkthrough Step 8 wire and S294 commit message) consumes `GET /api/quotes?status=approved` and stages those rows into `quote_intake_log`. If a future ABERP-side flow re-emits a status writeback as part of a recovery / partial-state-reconciliation pass — e.g. on a daemon restart that re-reads the same row from a checkpoint — the storefront refuses with `forbidden_transition: approved is only settable by the customer accept POST`. The error string is a misdescription of the actual event (it's an idempotent re-write from the bearer-authenticated channel, not an attempt to BYPASS the typed-ACCEPT). ABERP's classifier (line 1366-1368) treats `http 403` at stage=post as Permanent; **the daemon will mark the row Permanent and ledger a forensic-confusing payload that says "approved cannot be set here" when in fact the row was already approved**.
- **Recommended fix:** move the `from === to` noop check to the TOP of `checkTransition`, before the forbidden gates. Idempotent same-state is universally safe — the typed-ACCEPT path is gated on `from === 'quoted'`, not on what `to` is. Add a pin test `idempotent_same_state_approved_returns_noop_200` and `idempotent_same_state_quoted_returns_noop_200` in `status.spec.ts`.
- **Target session:** **S297 sweep (storefront-side)** — six lines moved + two pin tests. Bundle with the S295-sweep-tail items below.

## 2. Daemon stability — supervisor + classifier interplay

### F4 🟡 `StorefrontCredentialHandle::{set,clear}` silently swallow a poisoned RwLock

- **Evidence:** `apps/aberp/src/storefront_credential.rs:103-116`:
  ```rust
  pub fn set(&self, base_url: String, bearer: Zeroizing<String>) {
      if let Ok(mut guard) = self.inner.write() {
          *guard = Some(StorefrontCredentialSnapshot { base_url, bearer });
      }
  }
  ```
  On `Err(PoisonError)` (a thread panicked while holding the lock), `set` is a no-op. The PUT `/api/quote-intake/config` route at `serve.rs:16179` calls `set` after the toml + keychain write; on poisoned lock the seller.toml + keychain reflect the new URL but the in-memory daemon snapshot stays at the old value. **No log line, no error to the operator, no audit.** The catalogue-push daemon continues to push to the old URL until the next ABERP restart silently re-resolves.
- **Why it matters:** CLAUDE.md rule 12 — fail loud, not silent. The poison condition is rare but real (a panic anywhere in the catalogue-push code that's read-locking inner state would poison it for set/clear). The exact symptom is "operator changed the URL, daemon kept hitting the old one" — i.e. the bug S289 was written to fix, reappearing in a different shape.
- **Recommended fix:** unwrap via `unwrap_or_else(|p| p.into_inner())` (the pattern `runtime_discovery.rs:198` already uses for its test mutex) and emit a `tracing::warn!` naming the poison recovery. Alternatively, if the daemon should hard-fail on poison, panic and let the supervisor catch it (S286 panic-catch). The current swallow is the worst of both.
- **Target session:** S297 sweep — three lines.

### F5 🟡 `classify_failure` lowercases reason but NOT stage, despite the doc-comment claim

- **Evidence:** `quote_pricing_pipeline.rs:1331-1336`:
  ```rust
  /// Stays case-INSENSITIVE on the reason match by lowercasing both
  /// sides (matchings use lowercase literals); …

### F6 🟡 Classifier substring matches are unanchored and may false-positive

- **Evidence:** `quote_pricing_pipeline.rs:1338-1382` is a chain of `r.contains(…)` checks against case-folded reason. Rules of concern:
  - `r.contains("step file")` (line 1360, extract stage). Future error string `"Failed to read STEP file metadata (transient I/O error)"` lower-cases to `"failed to read step file metadata (transient i/o error)"` — substring match fires, classify Permanent, daemon silently never retries a genuinely transient I/O.
  - `r.contains("http 4")` (line 1372, post stage). Matches `http 4xx` but also `http 40-ish` if a future Display ever drops the curly braces. Loose.
  - `r.contains("not yet implemented")` (line 1338). Matches "STEP extraction not yet implemented" — fine. Also matches any new error like "feature X not yet implemented in v2 of the engine" at stage=price. Cross-talk risk if engine grows.
- **Why it matters:** these are belt-and-braces rules whose blast radius widens silently as the engine/wrapper Display strings evolve. The S290 memo's "default → Unknown" floor is the right posture, but the matchers should be either word-boundary-anchored OR enum-based on the typed-error variants the wrapper already enumerates (`aberp_cad_extract_wrapper::ExtractError` has 8 variants per S270 memo).
- **Recommended fix:**
  1. Short-term (S297 sweep): anchor each match with `\b` via a tiny regex set, or change to `.split_whitespace().any(|w| w == "step")` for the literal-word checks.
  2. Medium-term: classify on the typed `ExtractError` variant (`MaterialNotInCatalogue`, `SchemaVersionMismatch`, etc.) rather than on the `Display` string. Eliminates the false-positive class entirely.
- **Target session:** (1) S297, (2) S298+.

## 3. STEP extractor edge cases (S292)

F2 above is the headline. Three more:

### F9 🟡 OCCT stderr is NOT silenced; only stdout fd is redirected

- **Evidence:** `extractors/step.py:58-78` redirects fd 1 (stdout) to `/dev/null` during `ReadFile` and `TransferRoots`. fd 2 (stderr) is left intact. OCCT writes some diagnostic messages to stderr (`Could not read file ...`, `Warning: skipped entity #N`).
- **Why it matters:** the wrapper crate parses stdout JSON only (verified by `aberp_cad_extract_wrapper::CadExtractor::extract`), so stderr noise is captured into the wrapper's `ExtractError::Stderr` variant only on non-zero exit. On a happy-path STEP read with warning-only diagnostics, the stderr lines land in the subprocess stderr handle but are otherwise dropped — invisible to the audit trail. A future error path that exits 0 with warnings then a subtle wrong-shape FeatureGraph (e.g. partial transfer) is undebuggable from the ledger.
- **Recommended fix:** silence fd 2 too, OR pipe both fds to a tmpfile and attach the tail to the ExtractError on non-zero exit only. The current "ignore stderr on success" posture is fine if `TransferRoots` is guaranteed to fail-loud, but neither the OCCT docs nor the OCP bindings make that guarantee.
- **Target session:** S298+ (not load-bearing; deferred-with-rationale).

### F10 🟡 STEP extractor has no file-size cap or extractor-side timeout

- **Evidence:** `extractors/step.py:139-180`. No size check on the input file. No timeout on `ReadFile` or `TransferRoots`. The PricingPipelineService's reqwest client has a 30s timeout (`quote_pricing_pipeline.rs:158-161`) but that's for HTTP; the actual `spawn_blocking` of the Python CLI is unbounded.
- **Why it matters:** the storefront's `BODY_SIZE_LIMIT=50MB` (S295 F19) caps customer upload at 50 MB. A 50 MB STEP with high entity count (think 100k-solid assembly trim-mistake) takes minutes to parse and exhausts the daemon's per-cycle batch. While the `MAX_JOBS_PER_CYCLE: u32 = 5` (line 97) limits damage, one slow-STEP job stalls the row's transition for the whole subprocess time. No way for the operator to cancel from the SPA.
- **Recommended fix:** wrap the subprocess in a `tokio::time::timeout(Duration::from_secs(120), …)` and on timeout classify the failure as Permanent with reason `"STEP extraction exceeded 120s timeout (file too complex or assembly mistake)"`. Operator-action-required is the right verdict.
- **Target session:** S297 sweep — small change, real safety belt.

### F11 🟢 OCCT progress is silenced via OS fd redirect, NOT Python `redirect_stdout`

The S292 memo correctly diagnoses that `contextlib.redirect_stdout` can't reach C++ writes. `_silence_stdout_fd` (line 58-78) does the right thing — `os.dup(1)`, `os.dup2(devnull, 1)`, restore in `finally`. The fd-restore is exception-safe. ✅

### F12 🟢 Assembly handling fails loud with explicit error, not silent first-solid pick

`extractors/step.py:141-150` rejects 0-solid and N>1-solid shapes with classifier-friendly messages. The classifier rule at line 1360 catches both. No "pick the largest" or "sum the volumes" silent fallback that would silently surprise the operator. ✅

### F13 🟢 NotImplementedError gracefully degrades when OCP wheel is absent

`extractors/step.py:131-137` is the load-bearing escape hatch: a daemon running without the `[step]` extra emits "STEP extraction not yet implemented in this build" → classifier "not yet implemented" rule → Permanent. The SPA badge tells the operator to run `pip install -e '.[step]'`. The c1cf32 forensic pin still classifies Permanent via the same rule (per S292 memo). ✅

    local key="$1"
    if command -v jq >/dev/null 2>&1; then
      jq -r --arg k "${key}" '.[$k] // empty' "${DISCOVERY_FILE}"
    else
      python3 -c "import json,sys; print(json.load(open('${DISCOVERY_FILE}')).get('${key}',''))"
    fi
  }
  ```
  The python fallback does shell-interpolation into a python string literal. Today all callers pass closed-vocab literals (`base_url`, `tls_fingerprint`, `tenant`), so the key is safe. But if a future caller ever passes a user-controlled string or a key with a `'` in it, this is a shell-injection-via-key.
- **Why it matters:** dormant today; lethal if the launcher gains an `--extract <key>` flag. CLAUDE.md rule 8 — read before you write. The current code reads from a file ABERP writes; the python fallback assumes the key never has a quote. Defensive coding wins.
- **Recommended fix:** pass the key via argv: `python3 -c "import json,sys; print(json.load(open(sys.argv[1])).get(sys.argv[2],''))" "${DISCOVERY_FILE}" "${key}"`. Two seconds of work; eliminates the injection class.
- **Target session:** S297 sweep — three lines.

### F17 🟢 dev-test.sh refuses `tenant=prod` and self-syntax-checks via `bash -n`

`run/dev-test.sh:64-68` self-syntax-checks before running (mirrors `run_desktop.sh` PR-55 pattern). Line 103-106 hard-refuses `tenant=prod`. Storefront-port stragglers from a prior run get killed cleanly (line 250-259); ABERP-port collision fails hard (line 240-247) because there's a real process the operator should investigate. Discovery-file wait is bounded to 30s with a per-second poll. The cleanup trap covers SIGTERM/SIGKILL escalation per pid + a belt-and-suspenders `lsof` on storefront port. ✅

### F18 🟢 Runtime discovery file leaves bearer in keychain, not on disk

`apps/aberp/src/runtime_discovery.rs:84-89` carries only the keychain *pointer* (`<service>.<account>`) — the bearer never touches disk. ## 7. Boot-check coverage (S295 F8/F19)

### F25 🟡 BODY_SIZE_LIMIT boot-check runs once per process; reload-without-restart silently re-arms it

- **Evidence:** `ABERP-site/src/hooks.server.ts:35-46`:
  ```ts
  function runBootChecksOnce(): BootCheckResult {
      if (bootVerdict !== null) return bootVerdict;
      bootVerdict = runBootChecks();
      …
  }
  ```
  Cached at module scope. SystemD `systemctl reload` (vs `restart`) re-reads `Environment=` lines but Node's process doesn't restart, so `bootVerdict` stays cached at the boot-time value.
- **Why it matters:** the troubleshooting docs in the walkthrough say `sudo systemctl restart aberp-site` — correct. But an operator hand-editing `/etc/aberp-site.env` and running `reload` instead would (a) get new env in Node? No, systemd reload doesn't re-exec, so env stays old. So the cache is consistent with what the process sees. **Not actually a bug** — but the worry is real if anyone ever moves to a hot-reload posture. Tag-of-paranoia.
- **Recommended fix:** none beyond a comment in `boot-checks.ts` naming the cache invariant ("env is captured once at module load; systemctl reload does not bust the cache, which matches Node process env semantics"). Low-priority.
- **Target session:** S298+ deferred-with-rationale.

### F26 🟡 `checkRelayEnv` doesn't validate the operator email format

- **Evidence:** `ABERP-site/src/lib/server/boot-checks.ts:52-70`. `isPresent(operator)` only checks non-empty trim. A typo like `ABERP_SITE_OPERATOR_EMAIL=noemail` passes the boot check; relay-time send fails late (SMTP rejects bad-format envelope sender).
- **Why it matters:** the customer-CC behavior in `email.ts:317-321` adds the operator address to every email. A garbage value means every email-relay attempt 4xx's at the SMTP layer; the audit row records `email.relay_failed`; the customer never gets the priced email. The boot check exists precisely to catch this earlier.
- **Recommended fix:** add a minimal `/^[^\s@]+@[^\s@]+\.[^\s@]+$/` test (same regex as `+server.ts:46`) and surface a `F8-format` problem in the boot-check report. Two lines.
- **Target session:** S297 sweep.

### F27 🟡 ABERP has no boot-check for `ABERP_SISTER_SERVICE_BASE_URL` URL well-formedness

- **Evidence:** `apps/aberp/src/storefront_credential.rs:150-155` returns the trimmed env value with NO validation:
  ```rust
  pub fn read_sister_service_base_url_override() -> Option<String> {
      std::env::var(SISTER_SERVICE_BASE_URL_ENV)
          .ok()
          .map(|s| s.trim().to_string())
          .filter(|s| !s.is_empty())
  }
  ```
  Comment at line 147-149 says "bad URLs surface at the daemon's HTTP call, matching how toml + SPA-saved URLs are treated today (one validation path, not two)" — defensible philosophy.
- **Why it matters:** symmetric to the storefront F8 pattern. The dev-test.sh launcher sets this to a literal URL it controls; a future operator setting `ABERP_SISTER_SERVICE_BASE_URL=localhost:5173` (no scheme) gets a runtime daemon error 15 minutes after boot when the first push cycle fires. The one-shot INFO log at `serve.rs:1466-1471` names the override but doesn't validate.
- **Recommended fix:** **don't add validation here** (the comment's reasoning is sound). DO add a debug-mode assertion: when `ABERP_DEV_MODE=1` AND the URL fails `url::Url::parse`, emit a WARN at boot. Belt-and-braces; symmetric with `emit_dev_mode_prod_url_warning`. Minor.
- **Target session:** defer-with-rationale (philosophy is defensible; medium-low value).

### F28 🟢 `ABERP_INTERNAL_BASE_URL` vs `ABERP_SISTER_SERVICE_BASE_URL` naming is intentional

`ABERP_INTERNAL_BASE_URL` is the **storefront's** view of ABERP (set in Lightsail's `/etc/aberp-site.env`). `ABERP_SISTER_SERVICE_BASE_URL` is **ABERP's** view of the storefront (set by dev-test.sh on the ABERP side).
