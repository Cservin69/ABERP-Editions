# ADR-0098 — v0.2.6 deferred scope (read() coherence follow-through)

**Status:** Deferred to v0.2.6 (recorded at the v0.2.5 cut, 2026-07-02).
**Parent:** `adr/0098-editions-daemon-path-durability-and-recovery-guard-coherence.md`
(§ *v0.2.5 finalization (2026-07-02)*).

## Why this doc exists

v0.2.5 finalized ADR-0098 with **Option 1**: `Handle::read()` hands out a
`duckdb::Connection::try_clone` of the ONE shared instance (coherent reads — a
`try_clone` replays the live writer's WAL, so a `read()` observes every committed
write). This removed the Session-C2 F5 separate read-only instance, which was proven
to cause pervasive post-checkpoint stale reads (the `avl revoke→list` failure; four
group-A route modules red).

Option 1 accepts one narrow, deliberate gap for v0.2.5 — **Gap-2b: the loss of the
DB-level write-on-read fail-loud** — and defers two items to v0.2.6. This doc is the
standing record of that deferral so it cannot be quietly forgotten or allowed to grow.

## Deferred item (a) — compile-time write-on-read guard  [closes Gap-2b]

**Problem.** A `try_clone` is read-WRITE. A write *mis-routed* through `read()` (an
accidental `INSERT`/`UPDATE`/`CREATE` on a read connection) now succeeds silently —
committing to the shared instance but bypassing the `WriteGuard` post-commit hook
(lockstep mirror + debounced durable checkpoint). Under F5 this failed loud at the DB
layer (`AccessMode::ReadOnly` rejected it); Option 1 trades that runtime guard away.

**Why it is narrow (acceptable for v0.2.5).** All real writes already flow through
`Handle::write()`. The Session-C2 **class-1 audit** identified the known write-on-read
sites and routed them through `write()` (kept on the branch). `connection_is_read_only`
is retained (now inert on the read clone) and still guards the idempotent `ensure_schema`
DDL. So the residual is an *accidental future* mis-route, not a known live one.

**v0.2.6 fix — a compile-time read-guard type (stronger than F5, no separate instance).**
Have `Handle::read()` return a `ReadConn` newtype that exposes ONLY query methods
(`prepare`, `query_row`, `query_map`, …) and does **not** expose
`execute`/`execute_batch`/`transaction`. A mis-routed write then fails to **compile**
rather than failing at runtime (F5) or committing silently (v0.2.5). This re-closes
Gap-2b with a *stronger* guarantee and with **no** separate DuckDB instance — keeping
Option 1's coherence and tear-freedom. Sketch: `pub struct ReadConn(Connection);` with a
curated read-only surface; `read()` returns `ReadConn`; legitimate write call sites
switch to `write()` (each becomes a compile error that points at itself).

**Acceptance:** the write-on-read guard is compile-time; no `read()` caller can issue
DDL/DML; the try_clone coherence e2e stays green; the cut-gate's 10c-tryclone check
still holds.

## Deferred item (b) — residual `Ledger::open` / separate-audit-opener migration

**Problem.** ~180 runtime sites still open the live tenant DB independently of the
shared `Handle` — chiefly `Ledger::open` (audit appends), the per-mutation
`append_reopen` audit shape (avl's `create→revoke` pattern), and
`DuckDbBillingStore::open`. These are the operator-paced ERP request modules, CLI
one-shots, and serve.rs request-handler audit reads that Sessions B/C/C2 did **not**
migrate. They are lower-frequency than the daemons (operator- or CLI-paced), which is
why they were deferred rather than shipped-broken.

**The guardrail that makes the deferral safe — do NOT weaken it.** Cut-gate **CHECK
10i** freezes this residual surface in `tools/adr0098_c2_frozen_residuals.txt`: **139
runtime openers across 31 files** (as of the v0.2.5 cut). The gate FAILS if any listed
file EXCEEDS its frozen count, or if a NEW opener-bearing file appears unlisted. That is
what keeps a green D5 from ever again meaning "blind to most openers" (review F1–F4) and
what stops the deferred surface from silently growing before v0.2.6 migrates it.
**Keep CHECK 10i and its frozen list intact.**

**Highest-priority v0.2.6 targets** (per the manifest categories): the daemon-driven
modules reached at high frequency via `db.db_path()` — `incoming_invoices.rs` (9) +
`submission_queue.rs` / `drain_submission_queue.rs` (driven by the `ap_sync` backfill) —
then the remaining operator ERP modules, then the serve.rs request-handler audit reads
(30, `Ledger::open`). Migrate each onto `Handle::write()`/`read()` and drop its line from
the frozen manifest as it reaches 0 (CHECK 10h-style enforcement).

**Acceptance:** the frozen manifest shrinks monotonically toward 0; no file grows; the
two sanctioned residuals (the Handle's own boot open; the snapshot-EXPORT opener in
`take.rs`, CHECK 10g) remain the only non-Handle live opens.

## Out of scope (unchanged)

Prod stays frozen (ADR-0096); none of the above touches the prod line. The
`PROD_Defense_*` frozen edition snapshots are unaffected.

## R7 addendum (2026-07-06) — boot buyer-backfill re-fork closed + CHECK 10L

The v0.2.5 sweep (ADR-0098 R1–R6) FENCED but did not migrate
`restore_from_nav_outgoing::append_backfill_cycle_entry`. It stayed a RAW
`Connection::open` + a second `Ledger::open`/`sync_mirror` on the DB **path**,
spawned at boot (`serve.rs`) with `db_path` rather than the shared Handle. As a
separate DuckDB instance it read a **stale** ledger head, re-assigned an
already-taken `seq`, and rewrote the mirror from its own view — the 415/416
divergence the R1 boot guard refuses on. The `PRAGMA disable_checkpoint_on_shutdown`
fence (CHECK 10j) is orthogonal: it stops an implicit close-checkpoint, not a
stale-head `seq` collision or a rogue `sync_mirror`.

**R7 migrated (this session):**
* `append_backfill_cycle_entry` → shared `aberp_db::Handle` (`db.write()`), Handle
  threaded through `BackfillInputs` from the `serve.rs` boot spawn. Raw opener +
  `Ledger::open` + manual `sync_mirror` + fence pragma removed; the post-commit
  mirror is now the WriteGuard drop. This closes the exact boot re-fork.
* `incoming_invoices::change_status` (the sibling write-path residual) → same
  Handle migration, so an invoice status change can't fork next.
* Frozen ledgers updated: `restore_from_nav_outgoing.rs` 11→9,
  `incoming_invoices.rs` 7→4 (CHECK 10i count + CHECK 10k fingerprints).

**R7 gate — CHECK 10L** (`tools/cut_gate_db_isolation.sh`, teeth in
`cut_gate_negative_probes.sh`): freezes the *mirror-fork* class — any runtime fn
holding BOTH an independent live-DB opener AND a `sync_mirror` — in
`tools/adr0098_r7_mirror_fork_sites.txt` (16 sites; may only SHRINK), and pins
the two migrated seams opener-free + `sync_mirror`-free + `.write()`-routed. This
surfaces the fork-capable residual class permanently.

**Deliberately DEFERRED (flagged, conservative — not migrated this session):**
CHECK 10L is scoped to the fork-capable *write* class (opener + rogue
`sync_mirror`), NOT a blanket ban on every live-DB opener. A blanket ban would
red-out the ~136 residual openers this doc's item (b) deliberately defers
(protected by CHECK 10i/10k) and could not reach green-on-main. The remaining
path-based residual read helpers (`list_restored_missing_buyer`, `backfill_one_row`,
`list_restored`, the `incoming_invoices` reads, etc.) stay on the frozen ledger as
the v0.2.6 `Handle::read()` migration target — they are lower-risk (reads can go
stale but cannot re-assign a `seq` / rewrite the mirror). Migrate them next.
