# ADR-0098 ‚Äî v0.2.6 deferred scope (read() coherence follow-through)

**Status:** Deferred to v0.2.6 (recorded at the v0.2.5 cut, 2026-07-02).
**Parent:** `adr/0098-editions-daemon-path-durability-and-recovery-guard-coherence.md`
(¬ß *v0.2.5 finalization (2026-07-02)*).

## Why this doc exists

v0.2.5 finalized ADR-0098 with **Option 1**: `Handle::read()` hands out a
`duckdb::Connection::try_clone` of the ONE shared instance (coherent reads ‚Äî a
`try_clone` replays the live writer's WAL, so a `read()` observes every committed
write). This removed the Session-C2 F5 separate read-only instance, which was proven
to cause pervasive post-checkpoint stale reads (the `avl revoke‚Üílist` failure; four
group-A route modules red).

Option 1 accepts one narrow, deliberate gap for v0.2.5 ‚Äî **Gap-2b: the loss of the
DB-level write-on-read fail-loud** ‚Äî and defers two items to v0.2.6. This doc is the
standing record of that deferral so it cannot be quietly forgotten or allowed to grow.

## Deferred item (a) ‚Äî compile-time write-on-read guard  [closes Gap-2b]

**Problem.** A `try_clone` is read-WRITE. A write *mis-routed* through `read()` (an
accidental `INSERT`/`UPDATE`/`CREATE` on a read connection) now succeeds silently ‚Äî
committing to the shared instance but bypassing the `WriteGuard` post-commit hook
(lockstep mirror + debounced durable checkpoint). Under F5 this failed loud at the DB
layer (`AccessMode::ReadOnly` rejected it); Option 1 trades that runtime guard away.

**Why it is narrow (acceptable for v0.2.5).** All real writes already flow through
`Handle::write()`. The Session-C2 **class-1 audit** identified the known write-on-read
sites and routed them through `write()` (kept on the branch). `connection_is_read_only`
is retained (now inert on the read clone) and still guards the idempotent `ensure_schema`
DDL. So the residual is an *accidental future* mis-route, not a known live one.

**v0.2.6 fix ‚Äî a compile-time read-guard type (stronger than F5, no separate instance).**
Have `Handle::read()` return a `ReadConn` newtype that exposes ONLY query methods
(`prepare`, `query_row`, `query_map`, ‚Ä¶) and does **not** expose
`execute`/`execute_batch`/`transaction`. A mis-routed write then fails to **compile**
rather than failing at runtime (F5) or committing silently (v0.2.5). This re-closes
Gap-2b with a *stronger* guarantee and with **no** separate DuckDB instance ‚Äî keeping
Option 1's coherence and tear-freedom. Sketch: `pub struct ReadConn(Connection);` with a
curated read-only surface; `read()` returns `ReadConn`; legitimate write call sites
switch to `write()` (each becomes a compile error that points at itself).

**Acceptance:** the write-on-read guard is compile-time; no `read()` caller can issue
DDL/DML; the try_clone coherence e2e stays green; the cut-gate's 10c-tryclone check
still holds.

## Deferred item (b) ‚Äî residual `Ledger::open` / separate-audit-opener migration

**Problem.** ~180 runtime sites still open the live tenant DB independently of the
shared `Handle` ‚Äî chiefly `Ledger::open` (audit appends), the per-mutation
`append_reopen` audit shape (avl's `create‚Üírevoke` pattern), and
`DuckDbBillingStore::open`. These are the operator-paced ERP request modules, CLI
one-shots, and serve.rs request-handler audit reads that Sessions B/C/C2 did **not**
migrate. They are lower-frequency than the daemons (operator- or CLI-paced), which is
why they were deferred rather than shipped-broken.

**The guardrail that makes the deferral safe ‚Äî do NOT weaken it.** Cut-gate **CHECK
10i** freezes this residual surface in `tools/adr0098_c2_frozen_residuals.txt`: **139
runtime openers across 31 files** (as of the v0.2.5 cut). The gate FAILS if any listed
file EXCEEDS its frozen count, or if a NEW opener-bearing file appears unlisted. That is
what keeps a green D5 from ever again meaning "blind to most openers" (review F1‚ÄìF4) and
what stops the deferred surface from silently growing before v0.2.6 migrates it.
**Keep CHECK 10i and its frozen list intact.**

**Highest-priority v0.2.6 targets** (per the manifest categories): the daemon-driven
modules reached at high frequency via `db.db_path()` ‚Äî `incoming_invoices.rs` (9) +
`submission_queue.rs` / `drain_submission_queue.rs` (driven by the `ap_sync` backfill) ‚Äî
then the remaining operator ERP modules, then the serve.rs request-handler audit reads
(30, `Ledger::open`). Migrate each onto `Handle::write()`/`read()` and drop its line from
the frozen manifest as it reaches 0 (CHECK 10h-style enforcement).

**Acceptance:** the frozen manifest shrinks monotonically toward 0; no file grows; the
two sanctioned residuals (the Handle's own boot open; the snapshot-EXPORT opener in
`take.rs`, CHECK 10g) remain the only non-Handle live opens.

## Out of scope (unchanged)

`PROD_Defense_*` frozen edition snapshots are unaffected.
