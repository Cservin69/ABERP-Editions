## Summary

Total findings: **21** (🔴 3 / 🟡 11 / 🟢 7)

**What's solid (verified, not assumed):** zero phantom code surfaces — every type/path claimed by S355–S365 session reports exists exactly as claimed (`aberp_compliance::lot_heat::LotId`, `uid::Iuid`, `avl::DpasRating`, `incident::dod_72h_report_due_at_ms`, `UsDodCacProvider`, all 16 EventKinds). The F12 ritual is **clean 16/16 × 9 surfaces** — variant, `as_str`, `from_storage_str` (char-exact symmetric, fail-loud on unknown at `event_kind.rs:2644`), round-trip list, `ALL_KINDS`, both `extract_nav_xml` gates (exhaustive, no `_ =>`), per-family tests, prefix-distinctness tests. S364's `ALL_KINDS_COUNT == 103` is **correct**: enum has exactly 103 variants, `ALL_KINDS` lists 103 distinct entries, both compile-time tripwires present (`verify.rs:1078-1084`, `export_invoice_bundle.rs:949-955`). All `from_storage_str` impls across aberp-compliance return `Result` and error on unknown/empty input — no silent defaults anywhere. No production firing sites for any of the 16 kinds, matching every session's claim.

**What's broken:** the two flagship domain models are wrong against their own cited standards. S358's IUID Construct 1/2 definitions are **swapped** versus MIL-STD-130N — consistently, through doc, struct, ADR, and tests, so the test suite proves the wrong thing. S361's DPAS model hardcodes a closed 3-variant enum that cannot represent `DO-A1` — the canonical example rating in 15 CFR 700 itself — and its doc comment contradicts its own variant names within four lines. S360's banner helper knowingly drops the `SP-` specified-category prefix, so the one function whose purpose is "the compliant DoD banner" emits a non-compliant banner for CTI/EXPT.

**Top recommendations:** (1) swap the IUID construct definitions before any firing site exists — this is a 20-line fix today and a ledger-rewrite problem after S367+; (2) remodel `DpasRating` as rating-symbol × program-symbol before the partners column gets a writer; (3) emit `SP-` prefixes in `to_banner_str` for specified categories; (4) pick ONE operator-identity payload field name before the first compliance firing site lands.

## Cross-session patterns

1. **Internally-consistent wrongness is the failure mode of this batch.** F5 (IUID swap) and F13 (DPAS) are not sloppy code — doc, ADR, types, string pairs, and tests all agree with each other and disagree with the standard. The F12 ritual and round-trip tests verify *self*-consistency, which is necessary but cannot catch a misread of the source regulation. For compliance vocabulary, the review gate should require quoting the primary source (standard section, CFR cite, registry row) next to the model in the ADR — both F5 and F13 would have been caught by transcribing one example from the cited document (the DoD guide's Construct 1 example; 15 CFR 700's `DO-A1` example).
2. **The "no firing sites yet" posture is the saving grace — and the deadline.** Every 🔴 is cheap to fix today because zero ledger rows and zero writers exist. The moment S367+ wires the first firing site, F5/F13 become immutable-hash-chain corrections and F1 becomes a forensic-glob headache. Fix order matters more than usual.
3. **Operator identity needs one canonical payload key** (F1): ten distinct operator-field spellings across ADR-0073…0079 vs. `operator_user_id` in the two families that actually fire — and two *concepts* (Bearer subject vs. DigitalId handle) that no document yet distinguishes.
4. **Typed pairs exist; boundaries don't use them** (F3, F14): the crate consistently builds validated newtypes/fail-loud string pairs, and the app consistently adds raw VARCHAR columns with no path through them. The pattern only pays off if the future writers are *structurally forced* through the types — worth a standing rule: a compliance column may only be written via its `aberp_compliance` type.
5. **Two timestamp dialects on the compliance plane** (F4): epoch-ms in payloads, RFC3339 VARCHAR in columns, within the same sessions.
6. **Stub/mock hygiene is genuinely good** (F15, F9, WARN-on-construct everywhere, fail-loud parses, documented un-keyed digest) — the mock-everything boundary held; the residual risks are the unknown-env fallback and absent release guard (F18), both acceptable now but not after the first real backend.
